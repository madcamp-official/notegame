# Ninja Adventure — 제품 계약 v4

## 1. 제품 정의

`Ninja Adventure`은 코드리아(`WORLD_CODRIA`)의 시스템 붕괴를 조사하는 맵 중심 픽셀 어드벤처입니다. 넙죽이(`PROTAGONIST_NUPJUKYI`)는 관리자 키보드(`ARTIFACT_ADMIN_KEYBOARD`)를 깨우고, 세 단계 관리자 권한을 복구해 루트 시스템에 진입한 뒤 최종 배포와 결말을 선택합니다.

현재 NinjaAdventure의 `NinjaGreen`을 주인공 스프라이트로 임시 사용합니다. 이는 넙죽이의 표현 에셋일 뿐 제품명, 캐릭터 정체성이나 설정을 대신하지 않습니다.

## 2. 레퍼런스 이미지 경계

첨부 이미지는 인터페이스 감각만 참고합니다.

- 참고: 중앙의 큰 맵, 상단 상태, 우측 정보 레일, 하단 명령 덱, 따뜻한 목재·금속 픽셀 재질, 촘촘한 정보 위계
- 재설계: 패널 크기, 제목, 상태 이름, 아이콘 의미, 색상과 모든 상호작용 세부
- 사용 금지: 이미지의 세계, 인물, 적, 퀘스트, 문구, 재화, 지형 구성과 오브젝트 배치

레퍼런스에서 게임 컨셉, 지역 구성이나 시나리오를 추론하지 않습니다.

## 3. 한 번 생성하고 이동하는 월드

새 런은 160×160 월드를 한 번 결정적으로 생성합니다. 여섯 지역 축과 여섯 물리 바이옴, 도달 가능한 영역·경로·POI·배치 슬롯을 검증한 뒤 `layoutHash`와 함께 봉인합니다.

일반 턴, 저장·재개, Restore와 Undo는 타일, 영역 경계, 경로, POI나 슬롯을 재생성·이동하지 않습니다. 플레이 중에는 기존 슬롯을 활성화하고 엔티티가 봉인된 경로를 따라 이동합니다. LLM도 매 턴 맵을 구성하지 않습니다.

## 4. 고정 지역 축 6개

| ID | 제품상의 지역 정체성 |
| --- | --- |
| `REGION_BUG_FOREST` | 버그 숲 |
| `REGION_BUFFER_VILLAGE` | 버퍼 마을 |
| `REGION_DEADLOCK_CITY` | 데드락 시티 |
| `REGION_DATA_GRAND_LIBRARY` | 데이터 대도서관 |
| `REGION_LEGACY_CITADEL` | 레거시 성채 |
| `REGION_ROOT_SYSTEM` | 루트 시스템 |

지역 축은 서사·진행상의 장소 정체성입니다. 물리 바이옴은 별도 차원이며 지형, 팔레트, 이동 비용과 환경 규칙을 설명합니다.

1. `temperate_forest_field`
2. `river_wetland`
3. `arid_desert`
4. `frost_highland`
5. `subterranean_cavern`
6. `ancient_ruins`

모든 월드에는 여섯 축과 여섯 물리 바이옴이 각각 존재해야 합니다. 두 목록을 같은 enum, 같은 이름 또는 고정된 일대일 스테이지 순서로 취급하지 않습니다.

## 5. 캠페인 비트 9개

거시 진행은 정확히 다음 순서를 따릅니다.

1. 도착과 관리자 키보드 각성
2. 첫 붕괴 문제
3. 관리자 권한 I 획득
4. 관리자 권한 II 획득
5. 내부 관리자 통제 시스템이 붕괴 원인임을 확인
6. 기술 부채의 역류
7. 관리자 권한 III 획득
8. 루트 시스템 진입
9. 최종 배포와 결말

세 관리자 권한의 ID는 `ADMIN_ACCESS_LEVEL_1`, `ADMIN_ACCESS_LEVEL_2`, `ADMIN_ACCESS_LEVEL_3`입니다. 각 단계에는 서로 다른 영역에 있고 서로 다른 행동 맥락을 요구하는 획득 후보가 최소 두 개 있어야 합니다. 하나의 후보가 실패해도 런을 완료할 수 있어야 합니다.

루트 시스템은 세 권한 전체와 내부 관리자 통제 시스템에 관한 핵심 단서를 확보하기 전까지 잠깁니다. LLM 서술이나 높은 D20만으로 조건을 건너뛸 수 없습니다.

## 6. 입력과 턴 소비

서버가 저장하고 재생하는 권위 입력 envelope는 세 종류입니다.

| 입력 | 구조 | 규칙 |
| --- | --- | --- |
| `MOVE` | 검증 가능한 목적지 | 안전 이동은 D20·캠페인 턴을 쓰지 않음. 위험 이동은 안전 지점에 배치하고 조우를 활성화할 수 있음 |
| `USE_SKILL` | 기술과 검증 가능한 대상 | 실제 결과를 확정하며 의미 턴을 정확히 하나 소비 |
| `NARRATIVE_CHOICE` | 서버가 봉인한 대화·태도 선택 | D20 없이 서사 선택을 한 턴으로 확정 |

기술은 `COPY`, `DELETE`, `CONNECT`, `RESTORE`, `UNDO`, `SEARCH`, `SELECT_ALL` 일곱 개입니다. `COPY`는 복제, `DELETE`는 적 하나 공격, `SEARCH`는 선택 대상 조사, `UNDO`는 최근의 소비되지 않은 가역 턴 2회에 보상 연산 적용, `SELECT_ALL`은 주변 적 범위 공격을 담당합니다. MOVE는 기술이 아닙니다.

자유 입력은 네 번째 권위 입력이 아닙니다. `/v1/runs/:id/messages`가 문장 전체를 현재 위치·가시 대상·소유 아이템·제공된 목적지에 한정해 행동 제안으로 분류하고 Rule Engine이 다시 검증합니다. 대화·태도는 `NARRATIVE_CHOICE`, 판정 행동은 `USE_SKILL`로 정규화됩니다. Attack, Interact, Negotiate, Rest는 내부 정규화 행동이며 관리자 키보드 기술이나 별도 저장 입력 종류가 아닙니다. 불가능한 시도는 요청한 기계 효과 없이 구체적인 이유와 대안을 담은 서사 턴으로 기록될 수 있습니다. 모델을 사용할 수 없거나 출력이 잘못되면 같은 경계를 지키는 결정적 의미 분석 폴백을 사용합니다.

서버는 `USE_SKILL`을 `COMBAT`, `INVESTIGATION`, `NEGOTIATION`, `DEPLOYMENT` 중 하나의 맥락으로 분류합니다. 구조화 명령의 `playerNote`는 선택적 flavor이며 비어 있어도 행동할 수 있습니다. 자연어 원문이나 LLM 제안은 좌표, 합법성, 비용, D20, 권한이나 진행을 직접 결정하지 않습니다.

## 7. 규칙 권위와 기술 부채

Rule Engine이 경로, 점유, 대상 합법성, D20, 수정치, 피해, 자원, 권한, 핵심 단서, 상태 변화와 결말을 확정합니다. Undo는 최근 두 가역 턴의 역연산을 새 보상 이벤트로 적용하되 현재 턴, version, D20, 불변 사실과 봉인된 월드 geometry를 되감지 않습니다.

기술 부채는 수치와 인과 ledger를 함께 가집니다. 각 항목은 원인 턴, 기술·연산, 대상, 강제 우회 여부, 증감량, 지연 결과와 해소 근거를 기록합니다. 일반 성공은 기존 부채를 자동으로 낮추지 않습니다. 회복, 책임 수용, 자원 지불 또는 NPC 협력을 명시적으로 확정한 행동만 부채를 해소합니다.

## 8. 저장과 재개

다음 상태를 `layoutHash`와 함께 영속화합니다.

- `majorChoices`
- `regionOutcomes`
- `npcRelationships`
- `canonicalFacts`
- `unresolvedHooks`
- `abilityUsageHistory`
- `adminAccessAcquisitionHistory`
- `technicalDebtEntries`

또한 플레이어 위치·자원·인벤토리, 현재 비트, 목표, 조우, D20 판정, reversible action, 생성 계획과 LLM 감사 metadata를 저장합니다. 재개는 world/plan hash와 상태 checksum이 일치할 때만 허용합니다.

## 9. Gemini 사용 경계

- 기본 모델 `gemini-3.1-flash-lite`
- 최소 thinking level
- 필요한 ID와 상태만 담은 작은 context와 구조화 출력
- 스키마·의미 검증, 최대 1회 제한 재시도, 결정적 폴백
- 키는 서버 환경 변수 `GEMINI_API_KEY`에만 저장

Gemini는 확정 상태와 충돌하지 않는 짧은 장면 서술, 대사, 반응, 후속 hook과 에필로그를 제안할 수 있습니다. 좌표, 타일, 경로, 에셋 파일, 주사위, 피해, 보상, 관리자 권한, 핵심 사실, 부채 해소 또는 `endingId`를 확정할 수 없습니다.

Rule Engine이 누적 선택·지역 결과·관계·사실·부채로 허용 결말을 결정합니다. Gemini는 그 `endingId` 안에서 에필로그만 작성합니다.

## 10. UI 계약

- 주 목표 1개, 보조 목표 최대 2개
- 현재 상태에 맞는 추천 행동 2–3개
- MOVE와 일곱 기술을 명시하고, 사용할 수 없는 기술은 이유와 함께 비활성화
- 확정 전에 대상, 기술, 턴 소비 여부와 위험 표시
- 결과는 `판정 → 상태 변화 → 2–4문장 서술` 순서
- 안전 이동과 의미 행동의 턴 소비 차이를 명확히 표시
- 세 관리자 권한, 핵심 단서와 기술 부채 원인을 언제든 확인 가능
- 선택적 `playerNote`가 없다는 이유로 합법 행동을 막지 않으며 자유 입력은 접수·처리·거절·완료 상태를 표시

## 11. 완료 조건

- 같은 seed/version은 같은 160×160 `layoutHash`를 만들고 모든 턴·복구·재개가 이를 유지합니다.
- 고정 지역 축 6개와 물리 바이옴 6개가 독립적으로 존재하며 필수 위치가 도달 가능합니다.
- 정확히 9개 비트, 3개 관리자 권한과 단계별 2개 이상의 상이한 후보가 검증됩니다.
- 권한 3/3과 핵심 단서가 없으면 루트 진입이 거부됩니다.
- 안전 MOVE는 의미 턴을 쓰지 않고 유효한 USE_SKILL은 idempotent하게 한 턴만 확정합니다.
- 기술 부채와 필수 상태 이력이 저장·재개 후 동일하게 유지됩니다.
- Gemini 장애에도 규칙, 저장·재개와 결말 확정이 가능합니다.
- 레퍼런스 이미지는 인터페이스 감각만 남고 어떤 제품 콘텐츠도 복사되지 않습니다.
