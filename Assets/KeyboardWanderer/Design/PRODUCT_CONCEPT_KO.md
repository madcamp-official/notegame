# Keyboard Wanderer 제품 계약 v3

## 1. 제품 정의

Keyboard Wanderer는 하나의 봉인된 대형 픽셀 월드를 탐색하며, 키보드 명령과 자연어 의도로 매 런 다른 판타지 캠페인을 해결하는 30–50 의미 턴 게임입니다. Seed는 지형과 공간 관계를 결정하고, 검증된 LLM 기획은 기존 영역 안에서 장소의 의미, 인물, 갈등, 단서, 마일스톤과 마지막 선택을 구성합니다.

고정되는 것은 플레이 구조와 안전 규칙입니다. 런의 제목, 세계의 문제, 인물 관계, 비밀, 마일스톤의 의미와 결말 맥락은 고정 시나리오가 아닙니다. 현재 주인공 그래픽은 NinjaAdventure의 `NinjaGreen`을 사용합니다.

## 2. 레퍼런스 이미지 경계

첨부 이미지는 UI 감각만 참고합니다.

- 유지 가능: 중앙의 큰 맵, 상단 상태 바, 우측 로그·목표·D20 레일, 하단 행동 바, 따뜻한 목재·금속 픽셀 재질, 높은 정보 밀도
- 프로젝트에 맞게 재설계: 모든 제목, 장소, 인물, 적, 목표, 재화, 아이콘 의미와 정확한 패널 크기
- 복사 금지: 이미지의 마을·조우·수집 퀘스트, 캐릭터 배치, 문구와 고정 레벨 구성

이미지에서 세계관, 캠페인 순서 또는 바이옴 구성을 추론하지 않습니다.

## 3. 새 런 생성

캠페인 목록의 미리보기는 seed 기반 제목·톤·난이도 힌트만 계산하며 LLM을 호출하지 않습니다. 플레이어가 새 런을 확정하면 다음 파이프라인을 한 번 수행합니다.

1. seed, world generator version, campaign planner version과 목표 턴 수를 고정합니다.
2. 기본 160×160 타일 월드와 여섯 바이옴을 생성합니다.
3. 영역, 경로, 우회로, POI, 조우 slot과 finale 후보를 배치합니다.
4. 여섯 캠페인 역할을 서로 다른 적합 영역에 배정합니다.
5. 생성된 ID·공간 제약만 LLM에 제공해 캠페인 기획 후보를 받습니다.
6. 스키마, ID 참조, 역할 커버리지, 마일스톤 3개, 도달성, 턴 예산과 finale 구성을 검증합니다.
7. 실패한 필드는 결정적 보정 또는 제한 재시도로 복구합니다.
8. 확정된 world, campaign plan과 `layoutHash`를 원자적으로 저장한 뒤 플레이를 시작합니다.

플레이가 시작되면 타일, 영역 경계, 바이옴, 고정 POI와 경로를 봉인합니다. LLM과 턴 시스템은 발견 여부, 점유, 피해, 관계, 활성 조우 같은 검증된 희소 상태만 바꿀 수 있습니다.

## 4. 바이옴 6개

`BiomeDescriptor`는 시각 에셋, 팔레트, 이동 비용과 환경 규칙을 정의합니다. 모든 월드에 아래 여섯 종류가 존재해야 합니다.

| ID | 표현 범위 |
| --- | --- |
| `temperate_forest_field` | 숲, 들판, 수풀, 목재 길과 농경지 |
| `river_wetland` | 강, 호수, 폭포, 다리, 습지와 수변 정착지 |
| `arid_desert` | 사막, 모래, 건조 계곡과 풍화된 유적 |
| `frost_highland` | 설원 표현, 산악, 절벽, 고원 통로와 바람 지대 |
| `subterranean_cavern` | 동굴, 지하 던전, 광산과 지하 기록 공간 |
| `ancient_ruins` | 고대 사원, 탑, 성채와 오염된 설비 유적 |

바이옴은 고정 스테이지나 서사 순서가 아닙니다. 하나의 역할이 어느 바이옴에 배정되는지는 seed와 공간 조건에 따라 바뀝니다.

## 5. 캠페인 역할 6개

`CampaignRegionRole`은 어떤 고정 장소명이 아니라 캠페인에서 수행해야 할 기능입니다.

| 역할 | 필수 기능 |
| --- | --- |
| `ARRIVAL_CATALYST` | 플레이어가 세계의 현재 문제에 개입하게 되는 촉발 사건과 첫 선택 |
| `LOCAL_STAKES` | 한 지역과 주민에게 즉시 중요한 문제, 실패 비용과 첫 장기 결과 |
| `RELATIONSHIP_CONFLICT` | 서로 양립하기 어려운 인물·집단의 목표와 관계 선택 |
| `HIDDEN_TRUTH` | 앞선 사건의 해석을 바꾸는 검증 가능한 비밀과 증거 |
| `CONSEQUENCE_RETURN` | 이전 선택, 약속, 희생 또는 기술 부채가 다시 돌아오는 사건 |
| `FINAL_CONVERGENCE` | 핵심 인물·증거·마일스톤이 합류하는 마지막 공간 퍼즐과 결말 |

각 역할의 제목, 주민, 적, 목표, 문제 원인과 해결법은 런별 캠페인 기획이 생성합니다. 역할 커버리지와 순방향·우회 경로는 서버 검증기가 보장합니다.

## 6. 마일스톤 3개

진행 상태는 아래 세 토큰으로만 권위 있게 기록합니다.

1. `MILESTONE_TOKEN_1`
2. `MILESTONE_TOKEN_2`
3. `MILESTONE_TOKEN_3`

캠페인 기획은 각 토큰에 런별 표시명, 의미, 증거 조건, 획득 가능한 역할과 최소 두 개의 회복 경로를 연결합니다. 실패, 잘못된 역할, 반복 증거 또는 LLM 서술만으로 토큰을 지급하지 않습니다. `FINAL_CONVERGENCE` 진입 조건은 세 토큰과 이번 기획의 필수 증거를 규칙으로 평가합니다.

## 7. 탐색과 의미 턴

대형 월드의 이동 한 칸마다 캠페인 턴을 소비하지 않습니다.

- 안전 탐색: 발견된 통행 가능 타일·경로·POI 사이 이동. 서버가 경로와 점유를 검증하고 여행 시간과 발견 상태를 갱신하지만 D20과 의미 턴은 소비하지 않습니다.
- 위험 탐색: 위험도나 지름길 조건에 따라 조우를 열 수 있습니다. 조우가 열리는 것 자체와 그 안의 선택을 구분합니다.
- 의미 턴: 전투, 조사, 협상, 방어, 퍼즐, 복구, 확정된 키보드 배치처럼 세계 상태에 결과를 남기는 결정입니다.
- 조우 안의 `Move`: 자신이나 합법 대상을 전술 격자에서 재배치하는 주 명령이며 의미 턴을 소비합니다. 월드 탐색 이동과 별도입니다.

목표는 30–50 의미 턴 안에 완결된 한 런을 제공하는 것입니다. 남은 턴이 줄면 새 장기 갈등을 열지 않고 필수 단서와 회복 경로를 우선하며, hard cap에서는 LLM 없이도 유효한 비상 결말을 확정합니다.

## 8. 키보드 주 명령

| 명령 | 권위 효과 | 제한 예시 |
| --- | --- | --- |
| Move | 조우 안에서 자신 또는 합법 대상을 도달 가능한 칸으로 이동 | 거리, 지형, 경로, 점유 |
| Copy | 허용된 소형 객체나 효과의 불안정 사본 생성 | 고유·핵심 객체 금지, 비용과 부작용 |
| Delete | 삭제 가능한 객체나 임시 효과 제거 | 보호 대상과 영구 사실 금지 |
| Connect | 두 대상 사이에 허용된 관계 생성 | 관계 타입, 거리, TTL, 개수 |
| Restore | 최근 손상·삭제 상태를 권위 스냅샷으로 복구 | 복원 가능 필드, 대상, 비용 |
| Undo | 직전 가역 결과에 보상 이벤트 추가 | 턴·주사위·geometry·비가역 사실 유지 |

Attack, Interact/Investigate, Negotiate, Rest는 상황 행동이며 여섯 주 명령보다 시각적으로 보조 계층에 둡니다.

## 9. D20과 자연어 의도

플레이어는 명시적 대상 또는 목적지, 명령/상황 행동, 자유 입력 의도를 제출합니다. 서버는 원문을 보존하고 가장 가까운 합법적 시도로 정규화합니다. 불가능한 선언은 높은 주사위로도 가능해지지 않습니다.

- `mechanical score`: D20 + 권위 있는 수정치와 비용 대 난이도
- `intent alignment`: 원문이 선택한 행동·대상·목적과 얼마나 구체적으로 맞는지
- 결과 등급: 대실패, 실패, 부분 성공, 성공, 대성공
- 실패 진행: 단서, 비용, 관계 변화, 위험 증가, 우회로 또는 이후의 귀환 결과 중 검증된 효과를 남김

UI는 원본 D20, 수정치, 난이도, 정규화된 시도, 결과와 의도 차이를 한 화면에서 설명합니다.

## 10. Gemini 기획·서술 경계

- 기본 모델: `gemini-2.5-flash-lite`
- 키: 서버 환경 변수 `GEMINI_API_KEY`만 사용
- thinking budget: 0
- 런 생성은 작은 구조화 출력, 일반 턴은 필요한 상태만 담은 짧은 컨텍스트
- 최대 1회 제한 재시도 후 결정적 폴백

런 생성 시 허용되는 제안:

- 캠페인 제목, 설명과 톤
- 제공된 역할·영역·entity ID를 사용하는 beat, NPC, quest, ending과 area flavor
- 세 마일스톤의 런별 의미와 여러 획득 경로
- finale 구성 요소에 연결할 후보와 서사적 의미

턴 중 허용되는 제안:

- 확정 결과의 장면 서술, 대사, NPC 반응·기억 후보
- 기존 사실과 충돌하지 않는 소문·후속 실마리
- 서버가 제공한 entity와 slot에 대한 visual intent

금지되는 결정:

- 좌표, 타일, 바이옴, 출구, 경로와 고정 geometry 생성 또는 변경
- D20, 피해, 거리, 자원, 보상, 마일스톤, 지표와 결말 확정
- 존재하지 않는 ID, entity, area, POI, slot 또는 파일 경로 사용
- 프롬프트, 비밀키, 다른 사용자 데이터 접근

LLM은 스프라이트 파일을 고르지 않습니다. AssetResolver가 검증된 visual intent를 NinjaAdventure manifest의 실제 에셋으로 변환합니다.

## 11. 일반화된 마지막 공간 퍼즐

마지막 역할은 고정된 물건이나 고정 결말을 전제하지 않습니다. 캠페인 기획은 실제로 존재하는 entity를 다음 의미 구성 요소에 바인딩합니다.

- `anchor`: 지금 세계를 붙드는 핵심 대상
- `safeguard`: 파괴적 결과를 제한하는 보호 수단
- `memory`: 보존하거나 드러낼 기록·관계·증언
- `freedom`: 통제에서 벗어나려는 인물·집단·힘
- `threat`: 해결하지 않으면 남는 위험
- `passage`: 탈출, 이주, 전환 또는 귀환 경로
- `witness`: 선택의 결과를 기억하고 이어 갈 존재

검증된 finale recipe는 이 구성 요소의 배치, 연결, 보호, 제거와 누적 지표를 조건식으로 평가합니다. Rule Engine이 가능한 ending ID를 확정하고 Gemini는 그 범주 안에서 이번 런의 장소와 인물을 반영한 에필로그만 작성합니다. 필수 바인딩 누락이나 공급자 장애에는 생성 시 저장된 결정적 recipe가 사용됩니다.

## 12. 저장과 재개

반드시 영속화할 항목:

- seed, world/campaign version, `layoutHash`, 160×160 base 계약
- biome, area, route, POI, slot과 발견·희소 override
- 여섯 역할 배정, campaign plan, phase와 남은 필수 beat
- `MILESTONE_TOKEN_1`–`3` 상태와 획득 근거
- 플레이어 위치, HP, 자원, 인벤토리, 능력과 Restore/Undo ledger
- 세계 지표, 사실, 소문, 단서, 약속, 관계, 미해결 갈등과 귀환 결과
- NPC 역할, 위치, 생존, 관계와 기억
- 탐색 기록과 의미 턴 기록을 분리한 append-only event
- 원문 intent, 정규화된 시도, D20, 수정치, 난이도와 outcome
- 검증·거절된 LLM 제안, fallback, model/prompt/version/token/cost metadata
- finale 구성 요소 바인딩과 recipe version

재개 시 동일 입력으로 base world를 검증하고 `layoutHash`가 일치할 때만 희소 상태를 적용합니다. 불일치는 임의로 숨기지 않고 복구 상태로 격리합니다.

## 13. 완료 조건

- 같은 seed/version은 같은 160×160 `layoutHash`를 만들고, 일반 턴·복구·재개가 이를 바꾸지 않습니다.
- 여섯 바이옴과 여섯 캠페인 역할이 서로 독립적으로 존재하며 모든 필수 지점이 도달 가능합니다.
- 서로 구분되는 세 마일스톤과 최소 두 개의 회복 경로가 있고 마지막 역할이 올바르게 잠깁니다.
- 안전 탐색은 의미 턴을 쓰지 않고, 유효한 조우 행동은 idempotent하게 정확히 한 턴을 확정합니다.
- seed가 달라지면 제목, 인물 관계, 지역 문제, 비밀, 마일스톤 의미와 결말 맥락이 실질적으로 달라집니다.
- LLM이 없어도 월드 생성, 규칙 판정, 저장·재개와 hard-cap 종료가 가능합니다.
- `NinjaGreen`과 환경 에셋은 NinjaAdventure manifest를 통해 해석되며 픽셀 스냅을 유지합니다.
- UI는 레퍼런스 이미지의 구성·재질 감각만 사용하고 그 콘텐츠를 포함하지 않습니다.
- 서버, Unity, PostgreSQL 스키마, 테스트와 문서가 같은 v3 계약을 설명합니다.
